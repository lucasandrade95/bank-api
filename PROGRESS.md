# Progresso

- **2026-06-29** — Adicionei **request id (correlation id)** a toda requisição, fechando uma lacuna de observabilidade: até agora dava pra ver *quantas* operações aconteciam (métricas), mas não dava pra seguir os logs de *uma* chamada específica. Criei o `RequestIdFilter` (um `OncePerRequestFilter` com `@Order(HIGHEST_PRECEDENCE)`, logo roda antes até da segurança) que lê o cabeçalho `X-Request-Id` mandado pelo cliente/gateway — ou gera um `UUID` quando vem ausente/em branco — e (1) coloca o id no `MDC` do SLF4J, então toda linha de log da requisição já sai com ele (configurei `logging.pattern.level` pra imprimir `[%X{requestId:-}]` logo depois do nível), e (2) devolve o mesmo id no cabeçalho `X-Request-Id` da resposta, pro cliente referenciar a chamada num chamado de suporte. O filtro limpa o MDC num `finally` pra não vazar o id pra próxima requisição que reutilize a thread do pool. 2 testes MockMvc novos (gera um UUID quando o cliente não manda cabeçalho; ecoa o valor quando manda), batendo numa rota pública (`/actuator/health`) pra confirmar que o filtro roda independente de autenticação. Build verde: 42 testes.

- **2026-06-29** — Adicionei **travamento otimista (optimistic locking) no saldo da conta** para resolver o clássico *lost update* de sistema bancário: duas operações concorrentes (ex.: dois saques) que leem o mesmo saldo e gravam em sequência fariam uma sobrescrever a outra, perdendo dinheiro. Coloquei um campo `@Version` na entidade `Account` — o JPA passa a fazer `UPDATE ... WHERE id = ? AND version = ?` e a incrementar a coluna; a segunda transação atualiza **zero linhas** e o Hibernate lança `ObjectOptimisticLockingFailureException`. Mapeei essa exceção no `@RestControllerAdvice` para **409 Conflict** com o mesmo corpo de erro padrão (`ApiError`), com mensagem orientando o cliente a repetir a requisição. Escolhi otimista (em vez de bloqueio pessimista no banco) porque é mais escalável no caso comum de baixa contenção. Adicionei a migração Flyway `V2__add_account_version.sql` (coluna `version BIGINT NOT NULL DEFAULT 0`) para o profile `postgres`, que usa `ddl-auto: validate`. 3 testes novos (a `version` sobe a cada gravação; uma cópia *stale* gravada sobre uma versão mais nova falha com a exceção esperada; o handler traduz para 409) + 1 teste de migração conferindo que a coluna `version` é criada. Build verde: 40 testes.

- **2026-06-27** — Completei o **tratamento de erro centralizado** para dois caminhos que ainda escapavam do corpo padrão `ApiError`. Antes, um **JSON malformado/ausente** no corpo (ex.: `{ "ownerName":` sem fechar) ou um valor de tipo errado (ex.: `amount: "abc"`) fazia o Spring devolver o **corpo de erro dele**, fora do nosso padrão; e um **parâmetro de tipo inválido** (ex.: chamar `GET /api/v1/accounts/nao-e-um-uuid`, com um id que não é UUID) também caía fora do padrão. Adicionei dois handlers no `@RestControllerAdvice`: `HttpMessageNotReadableException` → **400** com mensagem "Corpo da requisicao ausente ou malformado", e `MethodArgumentTypeMismatchException` → **400** com `"<param>: valor invalido"`. Assim **toda** resposta de erro da API tem o mesmo formato (`timestamp`/`status`/`error`/`messages`) e nada de stack trace ou estrutura interna vaza pro cliente. 2 novos testes MockMvc (JSON quebrado e id que não é UUID, ambos conferindo o corpo padrão). Build verde: 36 testes.

- **2026-06-26** — Tornei o **extrato paginado**. Antes o endpoint `GET /api/v1/accounts/{id}/statement` carregava **todos** os lançamentos da conta de uma vez — não escala para uma conta com milhares de transações (custo de memória/banco e payload gigante). Agora ele aceita `?page=` e `?size=` e devolve um envelope `PageResponse` genérico (`content` + `page`/`size`/`totalElements`/`totalPages`/`last`), em vez de um array cru, para o cliente saber se há mais páginas. O `size` é validado (1–100) com `@Min`/`@Max`, então um pedido abusivo (`size=101`) ou inválido (`size=0`, `page=-1`) volta **400** pelo mesmo corpo de erro padrão — adicionei um handler de `ConstraintViolationException` no `@RestControllerAdvice`. A ordenação (mais recente primeiro) fica fixa no nome do método do repositório, não no `Pageable`, então o cliente controla só paginação e nunca consegue mudar a ordem do extrato. Atualizei os 5 testes de extrato para o novo formato e adicionei 2 testes (paginação respeitando `page`/`size` em 3 lançamentos e `size`/`page` inválidos devolvendo 400). Build verde: 34 testes.

- **2026-06-25** — Adicionei **observabilidade com Spring Boot Actuator + Micrometer**. Expus os endpoints `health`, `info` e `metrics` sob `/actuator`. Por segurança, `/actuator/health` e `/actuator/info` ficam **públicos** (servem de probe para load balancer/orquestrador e para o Kubernetes), enquanto `/actuator/metrics` exige token JWT — afinal métricas podem expor detalhes internos. Além das métricas técnicas que o Actuator já traz (JVM, HTTP, pool de conexões), criei uma **métrica de negócio**: o counter `bank.account.operations`, com tag `type` (`deposit`/`withdrawal`/`transfer`), incrementado dentro da transação a cada operação concluída — permite acompanhar o volume de operações por tipo num Grafana/Prometheus. 4 novos testes MockMvc: health público devolve `UP`, metrics sem token devolve 401, a métrica customizada aparece em `/actuator/metrics/bank.account.operations` e o counter de depósito sobe exatamente 1 após um depósito. Build verde: 32 testes.

- **2026-06-24** — Adicionei suporte a **PostgreSQL com migrações Flyway**, sem quebrar o fluxo simples de dev. O profile padrão segue com **H2 em memória** (Hibernate `ddl-auto: update`, Flyway desligado) para rodar e testar sem dependências externas. O novo profile `postgres` (`application-postgres.yml`) aponta para um Postgres real, usa `ddl-auto: validate` (o banco passa a ser **dono do schema**, o Hibernate só confere que as entidades batem) e deixa o **Flyway** aplicar a migração versionada `db/migration/V1__init_schema.sql` (tabelas `users`, `accounts`, `transactions`, com FK e índice por `account_id`). Credenciais vêm de variáveis de ambiente e há um `docker-compose.yml` que sobe o Postgres local. Para garantir que a migração não quebre, um teste (`FlywayMigrationTest`) executa o SQL num H2 em `MODE=PostgreSQL` e confirma que as três tabelas são criadas — roda no CI sem precisar de um Postgres de verdade. Build verde: 28 testes.

- **2026-06-23** — Adicionei autenticação JWT com Spring Security. As operações de conta (`/api/v1/accounts/**`) agora exigem `Authorization: Bearer <token>`; cadastro (`POST /api/v1/auth/register`), login (`POST /api/v1/auth/login`), health, Swagger e console H2 ficam públicos. O usuário (`AppUser`) guarda só o **hash BCrypt** da senha — nunca o texto puro. O `JwtService` emite e valida tokens HS256, com segredo e expiração vindos da configuração (`SECURITY_JWT_SECRET` em produção). Um `JwtAuthenticationFilter` (`OncePerRequestFilter`) lê o `Bearer` em cada requisição e popula o `SecurityContext`; a sessão é **stateless** (sem cookie/sessão no servidor). Acesso sem token ou com token inválido devolve **401** (entry point customizado com o mesmo corpo de erro padrão da API), e credenciais erradas no login também caem em 401 via `@RestControllerAdvice`. 9 novos testes MockMvc (registro, registro duplicado, senha curta, login ok, senha errada, rota protegida sem token, com token válido, com token lixo e health público) e os 18 testes de conta passaram a rodar autenticados com `@WithMockUser`. Build verde: 27 testes.

- **2026-06-19** — Adicionei transferência entre contas (`POST /api/v1/accounts/{id}/transfer`).
